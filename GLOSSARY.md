# Глоссарий

В данном документе приведены основные термины и сокращения, используемые в архитектурной документации проекта Athletica.

---

## 1. API (Интерфейс программирования приложений)
Набор правил и интерфейсов, позволяющий различным программным системам и сервисам взаимодействовать друг с другом.

## 2. ADR (Architecture Decision Record — запись архитектурного решения)
Документ, фиксирующий важное архитектурное решение, включая контекст, принятое решение и его последствия для системы.

## 3. Микросервисная архитектура
Архитектурный стиль, при котором система состоит из набора небольших независимых сервисов. Каждый сервис выполняет свою бизнес‑функцию и взаимодействует с другими через API.

## 4. Событийно‑ориентированная архитектура
Подход к построению системы, при котором сервисы взаимодействуют через события. Один сервис публикует событие, а другие сервисы подписываются на него и реагируют.

## 5. Kubernetes
Платформа оркестрации контейнеров с открытым исходным кодом, предназначенная для автоматического развертывания, масштабирования и управления контейнеризированными приложениями.

## 6. API Gateway
Компонент системы, который выступает единой точкой входа для клиентских запросов и маршрутизирует их к соответствующим сервисам. Также может выполнять аутентификацию, ограничение скорости запросов и мониторинг.

## 7. Наблюдаемость (Observability)
Способность системы предоставлять достаточную информацию о своём внутреннем состоянии через логи, метрики и трассировки, чтобы можно было анализировать её работу и находить причины проблем.

## 8. Метрики (Metrics)
Числовые показатели, собираемые во времени и характеризующие состояние и производительность системы.

## 9. Логи (Logs)
Записи событий, создаваемые приложениями или инфраструктурой. Используются для анализа работы системы и диагностики ошибок.

## 10. Трассировки (Traces)
Данные, показывающие путь запроса через несколько сервисов в распределённой системе.

## 11. Correlation ID (Идентификатор корреляции)
Уникальный идентификатор запроса, который передается между сервисами и используется для отслеживания одной бизнес‑операции во всей системе.

## 12. SLO (Service Level Objective — целевой уровень сервиса)
Целевое значение показателя работы сервиса, например доступность или время ответа.

## 13. SLA (Service Level Agreement — соглашение об уровне сервиса)
Формальное соглашение между поставщиком и пользователем сервиса, определяющее гарантируемый уровень качества обслуживания.

## 14. RTO (Recovery Time Objective — целевое время восстановления)
Максимально допустимое время восстановления системы после сбоя.

## 15. RPO (Recovery Point Objective — допустимая точка восстановления)
Максимально допустимый объём потери данных, измеряемый во времени.

## 16. CI/CD (Continuous Integration / Continuous Delivery — непрерывная интеграция и доставка)
Практика разработки, при которой изменения кода автоматически собираются, тестируются и разворачиваются.

## 17. CDN (Content Delivery Network — сеть доставки контента)
Географически распределённая сеть серверов, предназначенная для быстрой доставки статического контента пользователям.

## 18. IoT (Internet of Things — интернет вещей)
Физические устройства, подключённые к сети и способные собирать и передавать данные, например датчики, трекеры и другое оборудование.

## 19. Domain‑Driven Design DDD (предметно‑ориентированное проектирование)
Подход к проектированию программных систем, при котором архитектура строится вокруг предметной области бизнеса. Основное внимание уделяется моделированию доменов, бизнес‑правил и терминологии, используемой экспертами предметной области.

## 20. Bounded Context (ограниченный контекст)
Концепция Domain‑Driven Design. Чётко определённая граница внутри системы, в которой используется собственная модель данных, терминология и бизнес‑правила. Позволяет разделять сложную систему на независимые доменные области.

## 21. Event Broker (брокер событий)
Инфраструктурный компонент, обеспечивающий передачу сообщений и событий между сервисами. Используется для реализации событийно‑ориентированной архитектуры. Примеры: Kafka, RabbitMQ.

## 22. Real‑Time Processing (обработка в реальном времени)
Способ обработки данных, при котором информация анализируется и используется практически сразу после её получения, без значительных задержек.

## 23. Device Ingestion (приём данных от устройств)
Процесс получения, обработки и передачи данных, поступающих от внешних устройств или датчиков, например фитнес‑трекеров или других IoT‑устройств.

## 24. High‑Load System (высоконагруженная система)
Система, рассчитанная на обработку большого количества запросов, пользователей или данных в единицу времени.

## 25. Eventual Consistency (согласованность в конечном итоге)
Модель согласованности в распределённых системах, при которой данные в разных сервисах или узлах могут временно отличаться, но со временем становятся согласованными.

## 26. Distributed System (распределённая система)
Система, состоящая из множества взаимодействующих сервисов или компонентов, работающих на разных узлах сети и взаимодействующих через сеть.

## 27. Correlation ID Propagation (распространение идентификатора корреляции)
Практика передачи correlation‑id через все сервисы при обработке запроса. Позволяет связывать логи, метрики и трассировки, относящиеся к одной бизнес‑операции.

## 28. Cost-aware scaling (масштабирование с учётом стоимости инфраструктуры)
Подход к проектированию и масштабированию системы, при котором при увеличении нагрузки учитывается не только производительность, но и стоимость инфраструктуры. 
Цель — обеспечить баланс между производительностью системы и затратами на вычислительные ресурсы.

## 29. Pay-as-you-go (оплата по фактическому потреблению ресурсов)
Модель использования облачной инфраструктуры, при которой оплата производится только за фактически использованные ресурсы (вычислительные мощности, хранилище, сетевой трафик). 
Позволяет гибко масштабировать систему без необходимости заранее оплачивать фиксированные мощности.
