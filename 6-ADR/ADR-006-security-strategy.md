# ADR-006 — Стратегия безопасности системы

## Статус

Принято

---

## Контекст

Платформа Athletica обрабатывает чувствительные данные пользователей, включая:

- учётные записи;
- данные аутентификации;
- персональные данные профиля;
- данные спортивной активности;
- данные геолокации;
- телеметрию от устройств;
- данные коммерческих интеграций.

Система проектируется как распределённая платформа с микросервисной архитектурой (microservices architecture — микросервисная архитектура), доменной декомпозицией (bounded contexts — ограниченные контексты), асинхронным взаимодействием через Event Broker (RabbitMQ — брокер сообщений) и интеграцией с внешними сервисами и устройствами.

Согласно требованиям проекта, система должна обеспечивать:

- защиту персональных данных пользователей;
- безопасную аутентификацию и авторизацию;
- контроль доступа к сервисам и данным;
- устойчивость к несанкционированному доступу;
- соответствие юридическим и регуляторным требованиям разных стран.

Также в `risk.md` уже зафиксированы юридические и регуляторные риски, риски безопасности, а также риски использования сторонних API (интерфейс программирования приложений).

---

## Проблема

Без единой стратегии безопасности система столкнётся со следующими проблемами:

- утечка персональных данных;
- несанкционированный доступ к данным пользователей;
- компрометация учётных записей;
- рост рисков при интеграции с внешними сервисами и устройствами;
- невозможность доказать соответствие требованиям законодательства;
- высокая стоимость устранения инцидентов безопасности после запуска.

Так как Athletica является глобальной цифровой платформой, вопросы безопасности должны решаться архитектурно, а не точечно на уровне отдельных сервисов.

---

## Рассмотренные варианты

### 1. Децентрализованная безопасность на уровне каждого сервиса

**Описание:**

Каждый сервис самостоятельно реализует аутентификацию, авторизацию, правила доступа и защиту данных.

**Плюсы:**

- гибкость реализации для каждой команды;
- отсутствие единой точки ответственности.

**Минусы:**

- высокая вероятность расхождения подходов между сервисами;
- дублирование логики безопасности;
- сложность сопровождения;
- рост вероятности ошибок и уязвимостей;
- сложность аудита безопасности.

**Решение:**

Отклонено.

---

### 2. Полностью централизованная безопасность через один сервис

**Описание:**

Вся логика безопасности сосредоточена в одном центральном компоненте, включая аутентификацию, авторизацию и полный контроль доступа.

**Плюсы:**

- единая точка контроля;
- проще реализовать единые политики безопасности;
- проще проводить аудит.

**Минусы:**

- риск перегрузки одного компонента;
- возможная единая точка отказа;
- ограниченная гибкость для доменов с разными моделями доступа;
- сложность масштабирования всех аспектов безопасности через один сервис.

**Решение:**

Отклонено как слишком жёсткая модель.

---

### 3. Комбинированная стратегия безопасности

**Описание:**

Безопасность реализуется как комбинация:

- централизованной аутентификации и базовой авторизации через Auth Domain;
- доменного контроля доступа внутри сервисов;
- единой политики защиты API и данных;
- сквозных инфраструктурных механизмов безопасности.

**Плюсы:**

- единая модель идентификации пользователей;
- согласованность безопасности на уровне всей системы;
- гибкость доменов в реализации бизнес-правил доступа;
- лучшая масштабируемость;
- снижение риска дублирования и расхождения логики безопасности.

**Минусы:**

- требует более тщательного проектирования;
- повышает требования к архитектурной дисциплине;
- требует согласованных стандартов для всех команд.

**Решение:**

Принято.

---

## Принятое решение

### 0. Входной контур безопасности

Входной контур системы состоит из:

- NGINX (ingress — входной контур), выполняющего первичную фильтрацию, маршрутизацию и защиту трафика;
- API Gateway (шлюз API), реализующего аутентификацию, авторизацию и контроль доступа к API.

Такое разделение позволяет отделить инфраструктурный уровень безопасности от уровня бизнес-логики API.

### 1. Централизованная аутентификация

За аутентификацию отвечает **Auth Domain**.

Он управляет:

- регистрацией пользователей;
- входом в систему;
- управлением сессиями;
- токенами доступа;
- восстановлением доступа;
- базовыми политиками безопасности учётной записи.

---

### 2. Централизованная авторизация на базовом уровне

Auth Domain отвечает за:

- подтверждение личности пользователя;
- выпуск и проверку токенов;
- передачу базовых атрибутов доступа в другие домены.

---

### 3. Доменный контроль доступа

Каждый домен самостоятельно реализует **бизнес-правила доступа** к своим данным и операциям.

Например:

- Training Domain определяет, кто может просматривать тренировку;
- Social Domain определяет видимость публикаций и групп;
- Commerce Integration Domain определяет доступ к коммерческим предложениям;
- Analytics Domain ограничивает доступ к аналитическим данным.

---

### 4. Защита API

Внешний пользовательский и интеграционный трафик поступает в систему через NGINX (ingress — входной контур), после чего клиентские API-запросы обрабатываются через API Gateway (шлюз API).

Все внешние и внутренние API должны обеспечивать:

- проверку подлинности запросов;
- проверку прав доступа;
- валидацию входных данных;
- ограничение частоты запросов (rate limiting);
- аудит запросов;
- контроль доступа по ролям и контексту (RBAC — Role-Based Access Control — ролевая модель доступа);
- для внешнего взаимодействия обязательно использование HTTPS (защищённый HTTP) и TLS (Transport Layer Security — протокол шифрования);
- внутри доверенного контура (за NGINX и API Gateway) допускается использование внутренних протоколов при условии изоляции сети и недоступности сервисов извне;

---

### 5. Защита данных

Для защиты данных принимаются следующие правила:

- обязательное использование защищённых протоколов передачи данных (HTTPS — защищённый HTTP, TLS — Transport Layer Security — протокол шифрования транспортного уровня);
- защита чувствительных данных при хранении;
- разделение персональных и аналитических данных;
- минимизация объёма хранимых персональных данных;
- маскирование чувствительных данных в логах.

---

### 6. Безопасность интеграций и устройств

Интеграции с внешними API и устройствами должны выполняться через выделенные интеграционные компоненты.

Для них вводятся:

- контроль доверенных источников;
- проверка формата и валидности входящих данных;
- ограничение доступа к интеграционным API;
- аудит интеграционных вызовов.

---

## Обоснование решения

### 1. Соответствие архитектуре системы

Athletica проектируется как распределённая система с несколькими доменами и внешними интеграциями. Это требует единой модели аутентификации, но не означает, что вся логика доступа должна быть сосредоточена в одном месте.

---

### 2. Соответствие доменной декомпозиции

Согласно ADR-002, каждый домен имеет собственную зону ответственности. Следовательно, правила доступа к бизнес-данным также должны определяться на уровне домена.

---

### 3. Соответствие юридическим требованиям

В `risk.md` зафиксированы юридические и регуляторные риски, связанные с обработкой персональных данных и законодательством разных стран. Комбинированная модель безопасности позволяет:

- централизованно управлять идентификацией;
- гибко учитывать региональные требования;
- разделять персональные, операционные и аналитические данные.

---

### 4. Поддержка масштабирования

При росте системы централизовать всю безопасность в одном сервисе становится неэффективно. Комбинированный подход позволяет:

- масштабировать аутентификацию отдельно;
- масштабировать домены независимо;
- не блокировать развитие продукта из-за перегрузки одного security-компонента.

---

### 5. Снижение риска инцидентов

Чёткое разделение:

- аутентификация — в Auth Domain;
- бизнес-доступ — в доменах;
- инфраструктурная защита — на уровне платформы

уменьшает риск ошибок и упрощает аудит архитектуры безопасности.

---

## Последствия

### Положительные

- единая модель идентификации пользователей;
- согласованный подход к безопасности во всей системе;
- снижение дублирования логики;
- гибкость в реализации доменных правил доступа;
- лучшая подготовленность к юридическим и регуляторным требованиям;
- упрощение аудита безопасности.

### Отрицательные

- требуется строгая дисциплина в реализации security-правил;
- возрастает объём архитектурных соглашений между командами;
- появляется необходимость централизованного контроля стандартов безопасности;
- усложняется проектирование интеграций и доменных политик доступа.

---

## Ограничения и правила

При реализации стратегии безопасности должны соблюдаться следующие правила:

- аутентификация реализуется только через Auth Domain;
- домены не должны дублировать логику идентификации пользователя;
- запрещён прямой доступ к базе данных другого домена (no cross DB access — запрет прямого доступа к чужой базе данных);
- права доступа к бизнес-объектам определяются владельцем домена;
- чувствительные данные не должны логироваться;
- все внешние интеграции (входящие и исходящие) должны осуществляться исключительно через защищённые протоколы передачи данных (HTTPS — защищённый HTTP, TLS — Transport Layer Security — протокол шифрования транспортного уровня);
- внутри доверенного контура (за NGINX (ingress — входной контур) и API Gateway — шлюз API) допускается использование внутренних протоколов при условии изоляции сети и недоступности сервисов извне;
- все изменения ключевой security-модели оформляются отдельным ADR;
- безопасность должна проектироваться по принципу security by design (безопасность, закладываемая на этапе проектирования);
- обработка персональных данных должна учитывать требования законодательства и региональные ограничения.

---

### Политика обработки ошибок аутентификации и регистрации

Для внешнего API применяется HTTP-каноничная, но безопасная модель обработки ошибок:

- 400 Bad Request — ошибка формата запроса или некорректные входные данные;
- 401 Unauthorized — ошибка аутентификации (при этом причина ошибки не раскрывается);
- 503 Service Unavailable — недоступность внутренних сервисов или зависимостей.

При этом:

- внешнему клиенту не раскрываются точные причины отказа (например, существует ли пользователь или неверен пароль);
- используется единое сообщение об ошибке для сценариев аутентификации и регистрации;
- детализированная информация об ошибках фиксируется только во внутренних логах и системах наблюдаемости;
- используется correlation-id (идентификатор корреляции) для трассировки запросов и расследования инцидентов.

## Вывод

Для платформы Athletica принята комбинированная стратегия безопасности, в которой централизованная аутентификация и базовая авторизация сочетаются с доменным контролем доступа и инфраструктурными механизмами защиты. Это решение соответствует микросервисной архитектуре, доменной декомпозиции, требованиям безопасности и юридическим ограничениям глобальной платформы.
