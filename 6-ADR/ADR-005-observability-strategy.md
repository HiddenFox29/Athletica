

# ADR-005 — Стратегия наблюдаемости системы

## Статус

Принято

---

## Контекст

Платформа Athletica представляет собой распределённую систему (distributed system — распределённая система), состоящую из множества доменов (bounded contexts — ограниченные контексты), взаимодействующих через API (Application Programming Interface — интерфейс программирования приложений, через API Gateway — шлюз API) и Event Broker (RabbitMQ — брокер сообщений).

Система должна обеспечивать:

- высокую доступность;
- масштабируемость;
- устойчивость к сбоям;
- быстрое обнаружение проблем;
- возможность анализа причин инцидентов.

В условиях микросервисной архитектуры невозможно эффективно диагностировать проблемы без централизованной наблюдаемости.

Согласно требованиям (requirements.md), система должна обеспечивать контроль SLA (Service Level Agreement — соглашение об уровне сервиса) и быстрое восстановление после сбоев.

---

## Проблема

В распределённой системе без централизованной наблюдаемости возникают следующие проблемы:

- невозможно отследить путь запроса между сервисами;
- сложно выявить источник ошибки;
- отсутствует понимание состояния системы в реальном времени;
- увеличивается MTTR (Mean Time To Recovery — среднее время восстановления);
- затруднён анализ деградации производительности;
- сложно обнаруживать проблемы до их влияния на пользователей.

Необходимо определить стратегию наблюдаемости, которая обеспечит полный контроль над состоянием системы.

---

## Рассмотренные варианты

### 1. Локальное логирование без централизации

**Описание:**

Каждый сервис пишет логи локально без централизованного сбора.

**Плюсы:**

- простота реализации;
- отсутствие дополнительных компонентов.

**Минусы:**

- невозможно анализировать систему целиком;
- потеря логов при рестарте контейнеров;
- сложность расследования инцидентов;
- отсутствие корреляции между сервисами.

**Решение:**

Отклонено.

---

### 2. Только метрики (metrics — метрики)

**Описание:**

Система использует только метрики для мониторинга.

**Плюсы:**

- низкая нагрузка на систему;
- простота визуализации;
- удобно для отслеживания SLA.

**Минусы:**

- нет детальной информации об ошибках;
- невозможно восстановить цепочку событий;
- недостаточно для диагностики сложных проблем.

**Решение:**

Отклонено.

---

### 3. Полная observability (логи + метрики + трассировка)

**Описание:**

Система использует три основных сигнала наблюдаемости:

- логирование (logs — логи);
- метрики (metrics — метрики);
- трассировку (tracing — распределённая трассировка).

**Плюсы:**

- полный контроль над системой;
- возможность анализа инцидентов;
- обнаружение проблем на ранней стадии;
- соответствие современным стандартам распределённых систем.

**Минусы:**

- увеличение сложности инфраструктуры;
- рост объёмов данных;
- необходимость управления стоимостью хранения логов.

**Решение:**

Принято.

---

## Принятое решение

В системе Athletica реализуется стратегия observability (наблюдаемость), основанная на трёх компонентах:

### 1. Логи (Logs — логи)

- структурированное логирование в формате JSON;
- централизованный сбор логов;
- использование correlation-id для трассировки бизнес-операций;
- разделение логов по типам (application, access, audit, security).

---

### 2. Метрики (Metrics — метрики)

- сбор метрик по каждому сервису;
- контроль RPS, latency, error rate;
- мониторинг инфраструктуры (CPU, память, сеть);
- контроль SLA.

---

### 3. Трассировка (Tracing — распределённая трассировка)

- сквозная трассировка запросов;
- использование trace_id и correlation_id;
- отслеживание пути запроса через все сервисы;
- анализ задержек между сервисами.

---

## Обоснование решения

### 1. Соответствие архитектуре системы

Микросервисная архитектура требует распределённой наблюдаемости, так как запрос проходит через несколько сервисов.

---

### 2. Снижение MTTR

Наличие логов, метрик и трассировки позволяет быстрее находить и устранять проблемы.

---

### 3. Контроль SLA

Метрики позволяют отслеживать доступность и производительность системы.

---

### 4. Поддержка event-driven архитектуры

Event Broker (RabbitMQ — брокер сообщений) усложняет отслеживание событий, поэтому необходима трассировка и correlation-id.

---

### 5. Соответствие рискам

В risk.md указаны риски деградации системы и сложности диагностики. Observability снижает эти риски.

---

## Последствия

### Положительные

- быстрое обнаружение инцидентов;
- снижение времени восстановления;
- прозрачность работы системы;
- возможность проактивного мониторинга;
- повышение качества системы.

### Отрицательные

- рост инфраструктурной сложности;
- увеличение затрат на хранение логов;
- необходимость стандартизации логирования;
- дополнительные требования к разработчикам.

---

## Ограничения и правила

- все сервисы обязаны использовать структурированное логирование;
- все запросы должны содержать correlation-id;
- запрещено логировать чувствительные данные;
- все сервисы обязаны публиковать базовые метрики;
- трассировка должна покрывать критические бизнес-сценарии;
- логирование не должно влиять на производительность системы.

## Вывод

Стратегия observability (наблюдаемости), основанная на логах, метриках и трассировке, обеспечивает контроль над системой Athletica, снижает время обнаружения и устранения проблем и является обязательной для распределённой архитектуры.
