# ADR-003 — Выбор интеграционного подхода

## Статус

Принято

---

## Контекст

Платформа Athletica состоит из множества доменов (bounded contexts —
ограниченные контексты), которые должны взаимодействовать между собой и с
внешними системами.

К таким взаимодействиям относятся:

- пользовательские запросы (регистрация, тренировки, социальные действия);
- обработка событий (завершение тренировки, достижения, активность);
- интеграция с устройствами (IoT — Internet of Things — интернет вещей);
- аналитическая обработка данных;
- генерация уведомлений;
- интеграции с внешними системами.

Система должна обеспечивать:

- низкую связанность между доменами;
- масштабируемость;
- устойчивость к сбоям;
- возможность асинхронной обработки;
- гибкость развития интеграций.

---

## Проблема

При большом количестве доменов возникает вопрос выбора способа их
взаимодействия.

Если выбрать только синхронное взаимодействие, система становится сильно
связанной и уязвимой к сбоям.

Если выбрать только асинхронное взаимодействие, усложняется реализация
пользовательских сценариев, требующих мгновенного ответа.

Необходимо определить интеграционный подход, который обеспечит баланс между:

- скоростью отклика;
- надёжностью;
- масштабируемостью;
- сложностью реализации.

---

## Рассмотренные варианты

### 1. Полностью синхронное взаимодействие (REST API — Representational State Transfer — архитектурный стиль передачи данных)

**Описание:**

Все сервисы взаимодействуют через HTTP API (Application Programming Interface —
интерфейс программирования приложений).

**Плюсы:**

- простота реализации;
- понятность для разработчиков;
- удобство отладки;
- быстрый отклик для пользователя.

**Минусы:**

- высокая связанность сервисов;
- каскадные отказы;
- проблемы при высокой нагрузке;
- сложность масштабирования;
- зависимость от доступности всех сервисов.

**Решение:**

Отклонено.

---

### 2. Полностью асинхронное взаимодействие (Event-driven architecture — событийно-ориентированная архитектура)

**Описание:**

Все взаимодействия происходят через события с использованием брокера сообщений.

**Плюсы:**

- слабая связанность;
- высокая устойчивость;
- хорошая масштабируемость;
- возможность обработки больших потоков данных.

**Минусы:**

- сложность реализации;
- сложность отладки;
- eventual consistency (согласованность с задержкой);
- не подходит для пользовательских операций, требующих мгновенного ответа.

**Решение:**

Отклонено.

---

### 3. Комбинированный подход (API + Event-driven)

**Описание:**

Используется смешанная модель взаимодействия:

- синхронное взаимодействие через API (Application Programming Interface —
  интерфейс программирования приложений);
- асинхронное взаимодействие через Event Broker (брокер событий).

**Плюсы:**

- баланс между скоростью и устойчивостью;
- гибкость архитектуры;
- снижение связанности;
- возможность обработки событий в фоне;
- соответствие различным типам сценариев.

**Минусы:**

- усложнение архитектуры;
- необходимость управления двумя типами взаимодействий;
- требования к observability (наблюдаемости).

**Решение:**

Принято.

---

## Принятое решение

В системе Athletica используется комбинированный интеграционный подход:

### Синхронное взаимодействие (API)

Используется для:

- пользовательских запросов (регистрация, вход, получение данных);
- операций, требующих мгновенного ответа;
- межсервисных запросов в реальном времени.

---

### Асинхронное взаимодействие (Event Broker — RabbitMQ)

В качестве брокера сообщений выбран RabbitMQ (система очередей сообщений),
обеспечивающий маршрутизацию сообщений через exchange (обменники), поддержку
очередей, механизмов подтверждения доставки и повторной обработки сообщений.

Используется для:

- обработки событий тренировок;
- обновления аналитики;
- генерации рекомендаций;
- отправки уведомлений;
- обработки телеметрии устройств;
- социальных событий.

---

## Обоснование решения

### 1. Соответствие типам сценариев

Система содержит два типа операций:

- пользовательские (требуют мгновенного ответа);
- фоновые (могут выполняться асинхронно).

Комбинированный подход позволяет эффективно поддерживать оба типа.

---

### 2. Снижение связанности

Использование Event Broker позволяет доменам обмениваться событиями без прямой
зависимости друг от друга.

### 2.1 Выбор RabbitMQ как брокера сообщений

RabbitMQ выбран в качестве реализации Event Broker по следующим причинам:

- поддержка различных моделей маршрутизации сообщений (direct, topic, fanout);
- встроенные механизмы очередей, подтверждения доставки и повторной обработки;
- достаточная производительность для доменных событий системы;
- относительная простота эксплуатации по сравнению с более сложными решениями;
- соответствие требованиям системы без избыточной сложности.

---

### 3. Поддержка масштабирования

Асинхронная обработка позволяет масштабировать:

- аналитику;
- рекомендации;
- обработку событий устройств;
- социальные взаимодействия.

---

### 4. Устойчивость системы

Асинхронная модель позволяет системе продолжать работу даже при частичной
недоступности отдельных компонентов.

---

### 5. Соответствие архитектуре системы

Решение согласуется с:

- conceptual-architecture.md;
- development-roadmap.md;
- требованиями к масштабируемости и отказоустойчивости.

## Последствия

### Положительные

- гибкость архитектуры;
- высокая масштабируемость;
- снижение связанности;
- возможность обработки больших потоков данных;
- устойчивость к сбоям;
- использование RabbitMQ упрощает реализацию очередей и обработку событий

### Отрицательные

- усложнение архитектуры;
- необходимость внедрения распределённой трассировки;
- сложность отладки;
- необходимость управления консистентностью данных;
- необходимость настройки и сопровождения RabbitMQ (кластеризация,
  отказоустойчивость)

---

## Ограничения и правила

- синхронные вызовы используются только там, где это необходимо;
- асинхронные события являются основным способом междоменного взаимодействия;
- события должны быть идемпотентными (idempotency — идемпотентность);
- необходимо использовать correlation-id для отслеживания запросов;
- все интеграции должны быть описаны через контракты;
- необходимо использовать механизмы подтверждения доставки сообщений в
  RabbitMQ;
- необходимо использовать DLQ (dead letter queue — очередь ошибок) для
  обработки неуспешных сообщений;
- необходимо контролировать размер очередей и нагрузку на брокер сообщений.

## Вывод

Комбинированный интеграционный подход обеспечивает баланс между
производительностью, устойчивостью и масштабируемостью системы Athletica и
является оптимальным для распределённой архитектуры.
