# Список архитектурных решений (ADR — Architecture Decision Record)

Данный документ содержит список ключевых архитектурных решений, принятых при проектировании платформы Athletica.

Каждое решение оформлено в виде отдельного ADR (Architecture Decision Record — запись архитектурного решения) и содержит обоснование выбора с привязкой к бизнес-целям, функциональным и нефункциональным требованиям, а также рискам.

---

## Назначение ADR

ADR используются для:

- фиксации архитектурных решений;
- объяснения причин выбора решений;
- обеспечения прозрачности архитектуры;
- упрощения защиты архитектуры;
- возможности пересмотра решений в будущем.

Каждое решение связано с:

- бизнес-целями (vision-and-scope.md);
- функциональными требованиями (requirements.md);
- нефункциональными требованиями (requirements.md);
- рисками (risk.md);
- концептуальной архитектурой (conceptual-architecture.md);
- выбранными архитектурными опциями (architecture-options.md).

---

## Список ADR

### ADR-001 — Выбор архитектурного стиля системы

**Описание:**

Выбран микросервисный архитектурный стиль с разделением на домены  
(bounded contexts — ограниченные контексты) и использованием событийно-ориентированного взаимодействия  
(event-driven architecture — событийно-ориентированная архитектура).

**Связанные артефакты:**

- architecture-options.md  
- conceptual-architecture.md  
- requirements.md  

---

### ADR-002 — Декомпозиция системы на домены

**Описание:**

Система разделена на независимые доменные области  
(bounded contexts — ограниченные контексты):

- Auth Domain  
- Profile Domain  
- Training Domain  
- Social Domain  
- Gamification Domain  
- Recommendation Domain  
- Analytics Domain  
- Device Integration Domain  
- Commerce Integration Domain  

**Связанные артефакты:**

- conceptual-architecture.md  
- requirements.md  

---

### ADR-003 — Выбор интеграционного подхода

**Описание:**

Используется комбинированный подход взаимодействия:

- синхронное взаимодействие через API  
  (Application Programming Interface — интерфейс программирования приложений, через API Gateway — шлюз API);
- асинхронное взаимодействие через Event Broker  
  (RabbitMQ — брокер сообщений), зафиксировано в ADR-003.

**Связанные артефакты:**

- conceptual-architecture.md  
- requirements.md  
- risk.md  

---

### ADR-004 — Стратегия хранения данных

**Описание:**

Каждый домен управляет своими данными  
(database per service — отдельная база данных на сервис).

Для аналитики используется отдельный контур хранения и обработки данных.

**Связанные артефакты:**

- conceptual-architecture.md  
- requirements.md  
- risk.md  

---

### ADR-005 — Стратегия наблюдаемости системы

**Описание:**

Система реализует наблюдаемость  
(observability — наблюдаемость) через три компонента:

- логирование (logs — логи);  
- метрики (metrics — метрики);  
- трассировку (tracing — распределённая трассировка).  

**Связанные артефакты:**

- requirements.md  
- risk.md  

---

### ADR-006 — Стратегия безопасности

**Описание:**

Система реализует централизованную аутентификацию и авторизацию через Auth Domain,  
использует NGINX (ingress — входной контур) и API Gateway (шлюз API) как входной контур безопасности,  
а также обеспечивает защищённую передачу данных и контроль доступа.

**Связанные артефакты:**

- requirements.md  
- risk.md  
- conceptual-architecture.md  

---

### ADR-007 — Стратегия развёртывания и масштабирования

**Описание:**

Система проектируется как распределённая и масштабируемая,  
с возможностью горизонтального масштабирования и работы в нескольких регионах.

**Связанные артефакты:**

- development-roadmap.md  
- requirements.md  
- risk.md  

---

## Вывод

Список ADR фиксирует ключевые архитектурные решения системы и служит основой для защиты архитектуры проекта.

Каждое решение имеет обоснование и может быть использовано для ответа на вопросы проверяющих с опорой на уже существующие артефакты проекта.
