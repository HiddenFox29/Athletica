# ADR-004 — Стратегия хранения данных

## Статус

Принято

---

## Контекст

Платформа Athletica обрабатывает несколько различных типов данных:

- данные аутентификации и учётных записей пользователей;
- профили пользователей и пользовательские настройки;
- тренировочные данные и историю активности;
- социальные публикации и взаимодействия;
- события челленджей, достижений и лидербордов;
- телеметрию от устройств;
- данные аналитики и рекомендаций;
- данные интеграции с торговой экосистемой компании.

Система проектируется как микросервисная архитектура (microservices architecture — микросервисная архитектура) с доменной декомпозицией по bounded contexts (ограниченным контекстам). Это означает, что стратегия хранения данных должна соответствовать следующим требованиям:

- независимость доменов;
- возможность независимого масштабирования;
- снижение связанности между сервисами;
- поддержка аналитической обработки больших объёмов данных;
- устойчивость к отказам;
- соответствие требованиям безопасности и защиты персональных данных.

В рамках архитектурных решений также зафиксирован следующий технологический стек хранения и обработки данных:
- PostgreSQL (реляционная СУБД — система управления базами данных) как основное транзакционное хранилище;
- Object Storage (объектное хранилище, S3-compatible) для хранения медиа, файлов и резервных копий;
- Cache (кэш) для ускорения чтения и снижения нагрузки на транзакционные базы данных;
- Event Broker — RabbitMQ (брокер сообщений) для междоменного обмена событиями.

---

## Проблема

Если все домены платформы будут использовать единое хранилище данных, это приведёт к следующим проблемам:

- рост связанности между доменами;
- сложность независимого развития сервисов;
- трудности масштабирования отдельных частей системы;
- риск конфликтов между командами при изменении общей схемы данных;
- высокий риск деградации производительности при смешении транзакционных и аналитических нагрузок.

С другой стороны, полная изоляция данных без общей стратегии приведёт к:

- дублированию информации;
- усложнению аналитики;
- проблемам консистентности;
- сложности интеграции доменов между собой.

Необходимо выбрать стратегию хранения данных, которая обеспечит баланс между автономией доменов и возможностью сквозной аналитики и интеграции.

---

## Рассмотренные варианты

### 1. Единая централизованная база данных

**Описание:**

Все сервисы системы используют одну общую базу данных и общую схему.

**Плюсы:**

- простота стартовой реализации;
- единый источник данных;
- упрощение некоторых аналитических запросов.

**Минусы:**

- высокая связанность между доменами;
- сложность изменений общей схемы;
- невозможность независимого масштабирования доменов;
- смешение транзакционной и аналитической нагрузки;
- высокий риск блокировок и деградации производительности.

**Решение:**

Отклонено.

---

### 2. Общая база данных с разделением по схемам

**Описание:**

Сервисы используют одну физическую СУБД, но разделяют данные по логическим схемам.

**Плюсы:**

- частичная изоляция данных;
- проще администрирование на старте;
- ниже инфраструктурные затраты.

**Минусы:**

- сохраняется зависимость от одной физической базы;
- сложнее обеспечить независимое масштабирование;
- сохраняется риск обхода доменных границ через прямой доступ к данным;
- ограниченная автономность команд и сервисов.

**Решение:**

Отклонено как целевой вариант.

---

### 3. Database per Service (отдельная база данных на сервис)

**Описание:**

Каждый домен или сервис владеет своими данными и управляет своей схемой хранения. Для аналитики и рекомендаций используется отдельный аналитический контур.

**Плюсы:**

- низкая связанность между доменами;
- независимое развитие и миграции схем данных;
- независимое масштабирование по нагрузке;
- соответствие микросервисной архитектуре и bounded contexts;
- возможность выбора оптимального типа хранения для разных доменов.

**Минусы:**

- усложнение междоменной аналитики;
- необходимость управлять дублированием некоторых данных;
- необходимость управлять eventual consistency (согласованность с задержкой);
- усложнение операционного сопровождения.

**Решение:**

Принято.

---

## Принятое решение

Для платформы Athletica принимается стратегия **Database per Service (отдельная база данных на сервис)** с выделением отдельного аналитического контура данных.

Это означает:

- каждый домен владеет своими транзакционными данными;
- каждый домен самостоятельно управляет своей схемой хранения;
- прямой доступ одного домена к базе данных другого домена не допускается;
- обмен данными между доменами выполняется только через API (интерфейс программирования приложений, через API Gateway — шлюз API) или Event Broker (RabbitMQ — брокер сообщений);
- для аналитики, сравнений, рекомендаций и агрегированных отчётов используется отдельный аналитический слой.

В качестве основной технологии для транзакционных данных используется PostgreSQL, обеспечивающая надёжность, поддержку сложных запросов и соответствие требованиям к целостности данных. Медиа-данные, файлы и резервные копии выносятся в Object Storage (S3-compatible), а часто используемые данные могут кэшироваться с использованием Cache.

---

## Логика распределения данных по доменам

Некоторые домены (например, Profile Domain, Recommendation Domain, Social Domain) могут использовать Cache (кэш) для ускорения чтения и снижения нагрузки на PostgreSQL. При этом Cache не является источником истины и может содержать временно неконсистентные данные.

### 1. Auth Domain

Хранит:

- учётные записи пользователей;
- данные аутентификации;
- токены, сессии, данные восстановления доступа;
- сведения, связанные с безопасностью учётной записи.

### 2. Profile Domain

Хранит:

- профиль пользователя;
- спортивные цели;
- пользовательские настройки;
- интересы;
- уровень подготовки;
- ссылки на медиа (например, аватар пользователя, хранящийся в Object Storage);

### 3. Training Domain

Хранит:

- тренировочные сессии;
- маршруты;
- показатели активности;
- историю тренировок;
- результаты сравнений с собственными прошлыми тренировками;
- ссылки на файлы тренировок (например, треки), хранящиеся в Object Storage;

### 4. Social Domain

Хранит:

- публикации;
- связи пользователей;
- реакции и комментарии;
- данные групп и сообществ;
- ссылки на медиа-файлы публикаций, хранящиеся в Object Storage;

### 5. Challenge and Gamification Domain

Хранит:

- челленджи;
- достижения;
- таблицы лидеров;
- игровые показатели пользователя.

### 6. Device Integration Domain

Хранит:

- данные подключённых устройств;
- техническую информацию об интеграциях;
- входящую телеметрию устройств;
- историю синхронизации.

### 7. Recommendation Domain

Хранит:

- модели и правила рекомендаций;
- результаты генерации персональных рекомендаций;
- кэшированные персонализированные предложения.

### 8. Notification Domain

Хранит:

- шаблоны уведомлений;
- историю отправок;
- статусы доставки уведомлений.

### 9. Commerce Integration Domain

Хранит:

- данные интеграции с торговыми системами;
- региональные коммерческие предложения;
- ссылки на товары и предложения, релевантные пользователю.

### 10. Analytics Domain

Хранит:

- агрегированные данные активности;
- витрины данных для сравнительного анализа;
- обезличенные или подготовленные аналитические данные;
- данные для построения моделей рекомендаций и отчётности.

---

## Аналитический контур

Для платформы Athletica аналитика не должна выполняться напрямую на транзакционных базах доменов.

Поэтому принимается отдельный аналитический контур, который:

- получает события из доменных сервисов через Event Broker (RabbitMQ — брокер сообщений);
- агрегирует данные в отдельных хранилищах;
- используется для сравнительного анализа, рекомендаций, лидербордов, продуктовой аналитики и бизнес-отчётности.

Такой подход позволяет:

- не нагружать транзакционные сервисы аналитическими запросами;
- отделить оперативную обработку от аналитической;
- масштабировать аналитику независимо от пользовательских транзакций.

---

## Обоснование решения

### 1. Соответствие доменной декомпозиции

Решение напрямую следует из ADR-002, в котором система разделена на bounded contexts. Если домены независимы на уровне логики, они должны быть независимы и на уровне данных.

---

### 2. Поддержка масштабирования

Разные домены имеют разную нагрузку и скорость роста данных:

- Device Integration Domain генерирует поток телеметрии;
- Training Domain хранит историю активности;
- Social Domain хранит большое количество социальных событий;
- Analytics Domain работает с крупными агрегатами данных.

Разделение данных позволяет масштабировать их независимо.

---

### 3. Снижение риска изменений

Изменения схемы в одном домене не должны напрямую ломать другие домены. Стратегия Database per Service снижает риск каскадных изменений и ускоряет развитие отдельных частей системы.

---

### 4. Поддержка безопасности и юридических требований

Разделение данных позволяет:

- изолировать чувствительные данные аутентификации;
- отделять персональные данные от аналитических витрин;
- применять разные политики доступа и хранения;
- проще учитывать требования разных регионов к хранению пользовательских данных.

---

### 5. Соответствие roadmap платформы

Roadmap платформы предполагает постепенное расширение архитектуры по доменам. Стратегия хранения данных должна поддерживать этот рост, а не ограничивать его.

---

### 6. Выбор PostgreSQL как основной СУБД

PostgreSQL выбран в качестве основной СУБД по следующим причинам:

- поддержка сложных транзакций и строгой консистентности данных;
- развитые возможности индексации и оптимизации запросов;
- высокая надёжность и зрелость технологии;
- поддержка репликации и резервного копирования;
- широкая поддержка в экосистеме и совместимость с облачными провайдерами.

---

## Последствия

### Положительные

- домены становятся автономными на уровне логики и данных;
- проще масштабировать отдельные части системы;
- проще развивать и мигрировать схемы данных;
- уменьшается связанность между командами;
- аналитическая нагрузка выносится в отдельный контур;
- использование PostgreSQL обеспечивает надёжность хранения транзакционных данных;

### Отрицательные

- усложняется междоменная аналитика;
- возрастает роль Event Broker и интеграционных контрактов;
- появляются дублирование и eventual consistency (согласованность с задержкой) между доменами;
- возрастает операционная сложность сопровождения разных хранилищ;
- необходимость управления несколькими типами хранилищ (PostgreSQL, Object Storage, Cache);

---

## Ограничения и правила

При реализации стратегии хранения данных должны соблюдаться следующие правила:

- каждый домен владеет только своими данными;
- прямые SQL-запросы к данным другого домена запрещены;
- обмен между доменами осуществляется через API или события;
- аналитический контур не должен становиться транзакционным источником истины;
- чувствительные данные должны храниться и обрабатываться отдельно от обезличенных аналитических данных;
- любые изменения стратегии хранения данных оформляются через отдельный ADR;
- транзакционные данные должны храниться в PostgreSQL;
- медиа и файлы не должны храниться в транзакционных базах данных;
- доступ к данным должен осуществляться только через доменные сервисы;

## Вывод

Для платформы Athletica принята стратегия хранения данных **Database per Service (отдельная база данных на сервис)** с выделением отдельного аналитического контура. Это решение соответствует микросервисной архитектуре, доменной декомпозиции и требованиям к масштабируемости, безопасности и эволюционному развитию системы.
