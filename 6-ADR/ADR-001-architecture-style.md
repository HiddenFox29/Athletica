# ADR-001 — Выбор архитектурного стиля системы

## Статус

Принято

---

## Контекст

Платформа Athletica разрабатывается как глобальная цифровая система для пользователей, занимающихся спортом и использующих различные устройства отслеживания активности.

Система должна:

- поддерживать большое количество пользователей;
- обрабатывать данные от устройств (IoT — Internet of Things — интернет вещей);
- обеспечивать социальное взаимодействие;
- предоставлять персонализированные рекомендации;
- масштабироваться по мере роста нагрузки;
- обеспечивать высокую отказоустойчивость и наблюдаемость.

Согласно требованиям (requirements.md), система должна удовлетворять следующим нефункциональным требованиям:

- масштабируемость;
- отказоустойчивость;
- наблюдаемость (observability — наблюдаемость);
- безопасность;
- гибкость развития.

Также согласно рискам (risk.md):

- ожидается рост нагрузки;
- система будет распределённой;
- есть риски сложности интеграций;
- требуется поддержка множества доменов.

---

## Рассмотренные варианты

### 1. Монолитная архитектура (Monolithic Architecture — монолитная архитектура)

**Плюсы:**

- простота разработки на старте;
- единая кодовая база;
- низкий порог входа для команды;
- проще координация небольшой команды разработки.

**Минусы:**

- сложность масштабирования;
- высокая связанность компонентов;
- сложность внесения изменений;
- риск деградации при росте системы;
- ограниченные возможности параллельной разработки;
- при росте числа команд увеличивается количество конфликтов между зонами ответственности и изменениями в общей кодовой базе.

---

### 2. Модульный монолит (Modular Monolith — модульный монолит)

**Плюсы:**

- лучшее разделение ответственности по сравнению с монолитом;
- упрощённая эксплуатация;
- возможность постепенного перехода к микросервисам;
- позволяет распределить ответственность между несколькими командами при сохранении единой кодовой базы.

**Минусы:**

- ограниченное масштабирование;
- возможное нарушение границ модулей;
- рост сложности при увеличении функциональности;
- не оптимален для распределённых систем;
- требует высокой дисциплины команд, иначе границы модулей быстро размываются.

---

### 3. Микросервисная архитектура (Microservices Architecture — микросервисная архитектура)

**Плюсы:**

- независимое масштабирование сервисов;
- слабая связанность (loose coupling — слабая связанность);
- возможность независимого развития доменов;
- высокая отказоустойчивость;
- удобство интеграции с внешними системами;
- соответствие доменной декомпозиции (DDD — Domain-Driven Design — предметно-ориентированное проектирование);
- хорошо соответствует структуре большой организации, где разные команды могут независимо развивать свои домены.

**Минусы:**

- сложность разработки и эксплуатации;
- необходимость observability;
- сложность тестирования;
- увеличение инфраструктурных требований;
- требует зрелой организационной модели, при которой команды способны самостоятельно сопровождать свои сервисы.

---

## Принятое решение

Выбрана микросервисная архитектура (Microservices Architecture — микросервисная архитектура)  
с разделением на доменные области (bounded contexts — ограниченные контексты) и использованием:

- синхронного взаимодействия через API (Application Programming Interface — интерфейс программирования приложений);
- асинхронного взаимодействия через Event Broker (брокер событий).

---

## Обоснование решения

Выбор микросервисной архитектуры обусловлен следующими факторами:

### 1. Соответствие бизнес-целям

Система должна развиваться как экосистема, включающая:

- пользователей;
- устройства;
- социальные взаимодействия;
- аналитику и рекомендации.

Микросервисы позволяют развивать эти направления независимо.

### 2. Соответствие организационной модели компании

Компания имеет большой штат разработчиков, использующих разные технологии и работающих над несколькими направлениями продукта одновременно. 
Микросервисная архитектура лучше соответствует такой организационной модели, так как позволяет выделить независимые команды по доменам и снизить количество конфликтов при разработке.

### 3. Соответствие нефункциональным требованиям

| Требование | Как решается |
|---|---|
| Масштабируемость | горизонтальное масштабирование сервисов |
| Отказоустойчивость | изоляция сервисов |
| Наблюдаемость | внедрение логов, метрик и трассировки |
| Гибкость | независимое развитие сервисов |
| Интеграции | использование API и Event Broker |

---

### 4. Работа с IoT и событиями

Система обрабатывает:

- данные тренировок;
- события устройств;
- социальные события.

Использование Event Broker позволяет:

- обрабатывать события асинхронно;
- снижать связанность;
- масштабировать обработку данных.

---

### 5. Доменная декомпозиция (DDD)

Система уже разделена на домены:

- Auth;
- Profile;
- Training;
- Social;
- Gamification;
- Recommendation;
- Analytics;
- Device Integration;
- Commerce Integration.

Микросервисная архитектура естественно соответствует этой модели.

---

## Последствия

### Положительные

- высокая масштабируемость системы;
- гибкость разработки;
- независимость доменов;
- возможность работы с большими потоками событий;
- готовность к глобальному масштабированию.

---

### Отрицательные

- увеличение сложности системы;
- необходимость внедрения observability;
- повышение требований к инфраструктуре;
- сложность тестирования и отладки.

## Вывод

Выбор микросервисной архитектуры позволяет системе Athletica соответствовать бизнес-требованиям, нефункциональным требованиям и рискам проекта, обеспечивая масштабируемость, гибкость и готовность к развитию как глобальной цифровой платформы.
