# ADR-007 — Стратегия развёртывания и масштабирования

## Статус

Принято

---

## Контекст

Платформа Athletica проектируется как распределённая система (distributed system — распределённая система), работающая на глобальном уровне и обслуживающая большое количество пользователей.

Система включает:

- множество доменов (bounded contexts — ограниченные контексты);
- взаимодействие через API (Application Programming Interface — интерфейс программирования приложений) и Event Broker — RabbitMQ (брокер сообщений);
- интеграцию с внешними сервисами и устройствами (IoT — Internet of Things — интернет вещей);
- разные клиентские интерфейсы (mobile и web) с потенциально различными API-контрактами;
- обработку как транзакционных, так и аналитических нагрузок.

Согласно требованиям (requirements.md) и roadmap (development-roadmap.md), система должна обеспечивать:

- горизонтальное масштабирование;
- высокую доступность;
- отказоустойчивость;
- возможность быстрого выката новых версий;
- минимизацию простоев;
- поддержку глобального распределения нагрузки.

---

## Проблема

Без чёткой стратегии развёртывания и масштабирования система столкнётся со следующими проблемами:

- невозможность обрабатывать рост нагрузки;
- длительные простои при обновлениях;
- риск полной недоступности системы при сбоях;
- сложность масштабирования отдельных доменов;
- высокая стоимость эксплуатации при неэффективном масштабировании.

Необходимо выбрать стратегию, которая обеспечит баланс между:

- доступностью;
- масштабируемостью;
- стоимостью;
- сложностью управления инфраструктурой.

---

## Рассмотренные варианты

### 1. Вертикальное масштабирование (Vertical Scaling — вертикальное масштабирование)

**Описание:**

Увеличение ресурсов одного сервера (CPU, память, диск).

**Плюсы:**

- простота реализации;
- не требует сложной оркестрации;
- быстрый старт.

**Минусы:**

- ограничение по масштабированию;
- высокая стоимость при росте нагрузки;
- единая точка отказа;
- не подходит для глобальных систем.

**Решение:**

Отклонено.

---

### 2. Горизонтальное масштабирование без оркестрации

**Описание:**

Ручное добавление экземпляров сервисов и балансировка нагрузки.

**Плюсы:**

- лучше масштабируется, чем вертикальный подход;
- относительно простая модель.

**Минусы:**

- сложность управления большим количеством сервисов;
- отсутствие автоматизации;
- риск ошибок при ручном управлении;
- ограниченная отказоустойчивость.

**Решение:**

Отклонено.

---

### 3. Горизонтальное масштабирование с оркестрацией (Kubernetes — платформа оркестрации контейнеров)

**Описание:**

Сервисы развёртываются в контейнерах и управляются оркестратором.

**Плюсы:**

- автоматическое масштабирование;
- управление состоянием сервисов;
- автоматический перезапуск при сбоях;
- удобство деплоя и обновлений;
- поддержка multi-region архитектуры.

**Минусы:**

- высокая сложность инфраструктуры;
- требования к DevOps-команде;
- необходимость настройки мониторинга и безопасности.

**Решение:**

Принято.

---

## Принятое решение

Для платформы Athletica принимается стратегия:

### 1. Контейнеризация

Все сервисы упаковываются в контейнеры (Docker — контейнеризация приложений).

---

### 2. Оркестрация через Kubernetes

Используется Kubernetes (платформа оркестрации контейнеров) для:

- управления жизненным циклом сервисов;
- автоматического масштабирования;
- балансировки нагрузки;
- управления отказами.
- NGINX разворачивается как ingress-controller (контроллер входящего трафика) внутри Kubernetes для управления внешними запросами.

---

### 2.1 Входной контур и балансировка

В качестве входной точки системы используется NGINX (reverse proxy — обратный прокси и балансировщик нагрузки), выполняющий функции ingress (входного контура) и обеспечивающий:

- маршрутизацию входящих запросов;
- балансировку нагрузки между сервисами;
- терминацию TLS (шифрование соединений);
- базовую защиту и контроль трафика.

NGINX не реализует бизнес-логику API Gateway (шлюза API), а используется как инфраструктурный слой. Клиентские API (mobile и web) реализуются на уровне доменных сервисов и/или отдельного client API layer (слоя клиентских API).

---

### 3. Горизонтальное масштабирование

Каждый домен масштабируется независимо в зависимости от нагрузки.

---

### 4. Multi-region архитектура

Система развёртывается в нескольких регионах для обеспечения:

- отказоустойчивости;
- географической близости к пользователям;
- распределения нагрузки.

---

### 4.1 Геораспределение и региональные особенности

Система учитывает региональные особенности и бизнес-требования, включая:

- поддержку региональных промоакций;
- адаптацию контента и функциональности под регион;
- маршрутизацию пользователей в ближайший регион (Geo Routing — географическая маршрутизация);
- передачу региона как части контекста запроса.

Это позволяет реализовать бизнес-требование по вставке региональных промоакций без влияния на другие регионы.

При этом запросы от пользователей (mobile и web) и устройств проходят через NGINX, который выступает единой точкой входа и маршрутизации трафика в соответствующий регион и домены.

---

### 4.2 Репликация данных (Replication — копирование данных)

Для обеспечения доступности и отказоустойчивости используется репликация данных между регионами:

- репликация баз данных PostgreSQL (реляционная СУБД — система управления базами данных) между регионами;
- асинхронная репликация аналитических данных;
- синхронизация критичных данных (например, аутентификация и профиль);
- события через Event Broker (RabbitMQ — брокер сообщений) синхронизируются между регионами.

При этом:

- сохраняется принцип database per service (отдельная база данных на сервис);
- доменные границы не нарушаются;
- используется eventual consistency (согласованность в конечном итоге).

---

### 4.3 Отказоустойчивость и Disaster Recovery

Система поддерживает сценарии аварийного восстановления (Disaster Recovery — восстановление после сбоев):

- при отказе одного региона трафик перенаправляется в другой регион;
- используется балансировка нагрузки через DNS (Domain Name System — система доменных имён) и входной контур NGINX;
- Event Broker (RabbitMQ — брокер сообщений) обеспечивает повторную доставку событий;
- система может работать в деградированном режиме при частичной недоступности компонентов.

---

### 5. Стратегии выката (Deployment Strategies — стратегии развёртывания)

Используются:

- Blue-Green Deployment (развёртывание с переключением между двумя средами);
- Rolling Update (постепенное обновление);
- Canary Release (постепенный выпуск на часть пользователей).

---

### 6. Автоматизация CI/CD

Используется CI/CD (Continuous Integration / Continuous Delivery — непрерывная интеграция и доставка) для:

- автоматической сборки;
- тестирования;
- доставки изменений в production (рабочую среду).

---

## Обоснование решения

### 1. Соответствие архитектуре системы

Микросервисная архитектура требует независимого масштабирования сервисов.

### 1.1 Разделение ingress и API-логики

Разделение ролей между NGINX и клиентскими API позволяет:

- упростить инфраструктурный слой;
- избежать избыточного внедрения полноценного API Gateway;
- сохранить гибкость в развитии mobile и web API;
- реализовывать клиент-специфичную логику на уровне сервисов или отдельного слоя client API.

---

### 2. Поддержка роста нагрузки

Горизонтальное масштабирование позволяет добавлять ресурсы без остановки системы.

---

### 3. Отказоустойчивость

Multi-region архитектура позволяет системе продолжать работу при сбое одного региона. Также используется репликация данных PostgreSQL и multi-region стратегия с возможностью аварийного переключения между регионами (failover — переключение при сбое).

---

### 4. Минимизация downtime (время простоя системы)

Использование современных deployment стратегий позволяет обновлять систему без остановки.

---

### 5. Соответствие roadmap

Roadmap предусматривает постепенное масштабирование и расширение системы.

---

## Последствия

### Положительные

- высокая масштабируемость;
- отказоустойчивость;
- гибкость развёртывания;
- возможность быстрого внедрения изменений;
- поддержка глобальной инфраструктуры.

### Отрицательные

- сложность инфраструктуры;
- необходимость DevOps-экспертизы;
- рост стоимости поддержки;
- необходимость мониторинга и оптимизации ресурсов.

---

## Ограничения и правила

- все сервисы должны быть контейнеризированы;
- каждый сервис должен поддерживать горизонтальное масштабирование;
- инфраструктура должна поддерживать multi-region;
- деплой должен быть автоматизирован;
- изменения должны проходить через CI/CD pipeline;
- стратегия развёртывания должна минимизировать downtime;
- масштабирование должно учитывать cost-aware scaling (масштабирование с учётом стоимости инфраструктуры);
- любые изменения стратегии развёртывания оформляются через ADR;
- система должна поддерживать маршрутизацию пользователей между регионами;
- данные должны реплицироваться между регионами с учётом eventual consistency (согласованность с задержкой);
- критичные компоненты должны иметь стратегию Disaster Recovery;
- весь внешний трафик (mobile, web, устройства) должен проходить через NGINX как единую точку входа;
- взаимодействие между сервисами должно учитывать использование RabbitMQ как брокера сообщений;
- транзакционные данные должны храниться в PostgreSQL;

Стратегия развёртывания и масштабирования Athletica основана на контейнеризации, Kubernetes, горизонтальном масштабировании и multi-region архитектуре. Это обеспечивает высокую доступность, отказоустойчивость и готовность системы к глобальному росту.
