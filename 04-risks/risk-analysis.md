# Анализ рисков архитектуры и компромиссов

Данный документ содержит анализ рисков реализации архитектуры системы Athletica и принятых компромиссов. Анализ опирается на ранее выявленные риски (risk.md), а также на финальную архитектуру:
- microservices architecture (микросервисная архитектура);
- event-driven architecture (событийно-ориентированная архитектура);
- multi-region deployment (многорегиональное развёртывание);
- использование Databases (базы данных), Object Storage (объектное хранилище, S3-compatible) и Cache (кэш).

Связанные архитектурные артефакты:
- base-architecture.md;
- architecture-views.md;
- ADR-001, ADR-002, ADR-003, ADR-004, ADR-007;
- use-cases (01–05);
- risk.md.

---

## 1. Метод оценки рисков

Для каждого риска оцениваются:
- вероятность (низкая / средняя / высокая);
- влияние (низкое / среднее / критичное);
- трудозатраты на устранение;
- последствия игнорирования.

---

## 2. Основные риски архитектуры

### 2.1 Сложность микросервисной архитектуры

- Вероятность: высокая  
- Влияние: высокое  
- Трудозатраты: высокие  
- Стоимость решения: высокая

Описание:  
Большое количество сервисов увеличивает сложность разработки, тестирования и сопровождения.

Последствия:  
- увеличение количества ошибок;
- усложнение отладки;
- рост time-to-market (время вывода продукта на рынок).

Снижение риска:  
- единые стандарты API;
- централизованная документация;
- автоматизация CI/CD (непрерывная интеграция и доставка).

---

### 2.2 Event-driven архитектура и eventual consistency (согласованность с задержкой)

- Вероятность: высокая  
- Влияние: среднее  
- Трудозатраты: высокие  
- Стоимость решения: средняя

Описание:  
Асинхронные события приводят к временной несогласованности данных.

Последствия:  
- пользователь может видеть устаревшие данные;
- сложность тестирования и отладки.

Снижение риска:  
- идемпотентность операций;
- retry (повторные попытки);
- мониторинг событий.

---

### 2.3 Потеря или дублирование событий (Event Broker — брокер событий)

- Вероятность: средняя  
- Влияние: критичное  
- Трудозатраты: высокие  
- Стоимость решения: средняя

Описание:  
Сбой брокера может привести к потере или повторной доставке сообщений.

Последствия:  
- потеря данных;
- некорректная аналитика;
- дублирование операций.

Снижение риска:  
- durable queues (устойчивые очереди);
- DLQ (dead letter queue — очередь ошибок);
- идемпотентная обработка.

---

### 2.4 Риски хранения данных (Databases)

- Вероятность: средняя  
- Влияние: критичное  
- Трудозатраты: высокие  
- Стоимость решения: высокая

Описание:  
Сбой базы данных может привести к потере транзакционных данных.

Последствия:  
- потеря пользовательских данных;
- нарушение бизнес-логики;
- недоступность системы;
- репутационные потери;
- финансовые потери;

Снижение риска:  
- репликация баз данных;
- резервное копирование;
- изоляция доменов (database per service).

---

### 2.5 Риски Object Storage (S3-compatible)

- Вероятность: средняя  
- Влияние: среднее  
- Трудозатраты: высокие  
- Стоимость решения: средняя

Описание:  
Проблемы с объектным хранилищем могут привести к недоступности медиа.

Последствия:  
- отсутствие изображений и файлов;
- ухудшение пользовательского опыта;
- снижение вовлечённости пользователей;

Снижение риска:  
- репликация между регионами;
- резервное копирование;
- хранение только ссылок в DB.

---

### 2.6 Риски Cache (кэш)

- Вероятность: высокая  
- Влияние: среднее  
- Трудозатраты: высокие  
- Стоимость решения: средняя

Описание:  
Кэш может содержать устаревшие данные.

Последствия:  
- отображение неактуальной информации;
- неконсистентность UI (пользовательский интерфейс);
- рассинхронизация данных между Cache и DB (Database — база данных);

Снижение риска:  
- TTL (time-to-live — время жизни записи);
- инвалидация кэша;
- fallback (возврат) на DB (Database — база данных).

---

### 2.7 Multi-region сложность

- Вероятность: средняя  
- Влияние: высокое  
- Трудозатраты: высокие  
- Стоимость решения: средняя

Описание:  
Синхронизация данных между регионами усложняет архитектуру.

Последствия:  
- конфликты данных;
- задержки;
- увеличение стоимости;
- рост операционных затрат;

Снижение риска:  
- разделение критичных и некритичных данных;
- асинхронная репликация;
- чёткое определение источника истины.

---

### 2.8 Рост нагрузки и стоимости

- Вероятность: высокая  
- Влияние: высокое  
- Трудозатраты: высокие  
- Стоимость решения: средняя

Описание:  
Рост пользователей увеличивает нагрузку и стоимость инфраструктуры.

Последствия:  
- деградация производительности;
- рост затрат;
- финансовые потери при неконтролируемом росте инфраструктуры;

Снижение риска:  
- горизонтальное масштабирование;
- использование Cache (кэш);
- оптимизация хранения данных.

---

## 3. Компромиссы архитектуры

### 3.1 Consistency vs Availability

Выбран компромисс: eventual consistency (согласованность с задержкой).

Причина:  
- повышение масштабируемости;
- снижение нагрузки на систему.

---

### 3.2 Стоимость vs Надёжность

Выбран компромисс: multi-region + standby.

Следствие:  
- увеличение стоимости;
- повышение отказоустойчивости.

---

### 3.3 Простота vs Масштабируемость

Выбран компромисс: сложная микросервисная архитектура.

Причина:  
- гибкость;
- возможность роста системы.

---

## 4. Связь рисков с архитектурными решениями

Каждый из перечисленных рисков напрямую связан с принятыми архитектурными решениями:

- микросервисная архитектура (ADR-001) увеличивает гибкость, но повышает сложность системы;
- event-driven подход (ADR-003) приводит к eventual consistency и рискам обработки событий;
- стратегия хранения данных (ADR-004) определяет использование Databases, Object Storage и Cache и связанные с ними риски;
- multi-region архитектура (ADR-007) увеличивает отказоустойчивость, но добавляет сложность синхронизации и увеличивает стоимость;
- наблюдаемость (ADR-005) снижает риски диагностики и ускоряет реакцию на инциденты.

Данные риски также отражены и частично покрыты в следующих документах:
- base-architecture.md — инфраструктурные компоненты и схема развёртывания;
- architecture-views.md — представления системы и потоки данных;
- use-cases — конкретные сценарии, в которых проявляются риски;
- risk.md — исходный список рисков проекта.

## 5. Вывод

Архитектура системы является сложной, но оправданной.

Риски контролируемы за счёт:
- использования репликации;
- разделения данных;
- применения кэширования;
- мониторинга системы.

Выбранные решения обеспечивают баланс между:
- производительностью;
- надёжностью;
- стоимостью владения системой.
